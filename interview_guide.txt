OR
Can you tell me about a real time streaming project that you worked on?
As part of my professional experience, I had the exciting opportunity to lead the design of a cutting-edge data lake, where I was tasked with building a real-time ingestion architecture. In this project, I successfully designed a robust system that could efficiently ingest JSON events in real-time using API Gateway and Kinesis Firehose to S3.
The upstream events were generated by Google Pub/Sub and were seamlessly posted to our API Gateway, ensuring a smooth flow of data. To prioritize data security, we implemented mutual authentication, ensuring the integrity of the information.
Once the mutual authentication process was successful, the messages were directed to Kinesis, which efficiently transmitted the data to multiple Firehoses. These Firehoses, in turn, posted the data into two S3 buckets. One bucket served as an archive folder, while the other was specifically utilized by Snowflake for data ingestion.
In order to facilitate seamless data access for various downstream processes, we opted to use Kinesis Firehose. This strategic choice allowed us to cater to any future requirements for data by easily sending the same events to other downstream processes, such as Grafana.
For loading the data into Snowflake, we relied on the powerful capabilities of DBT. By utilizing DBT Cloud, we could efficiently run copy commands and design macros that ran copy commands for all the JSON events. The raw JSON data was directly loaded into the Raw layer, ensuring we maintained the original form of the data.
To further enhance data processing, we harnessed the capabilities of DBT Cloud as a powerful transformation tool, effectively implementing the required business logic. Scheduling DBT Cloud to run every 15 minutes allowed us to maintain real-time data access by regularly loading data into Snowflake.
Our Snowflake database was structured into three layers, each serving specific purposes. The Raw layer housed the data in its original format, without any alterations. The Staging layer came into play to flatten the JSON events into columnar-level data. This particular layer was structured as SCD2, ensuring all historical data was readily accessible for future use.
Lastly, the DWH (Data Warehouse) layer was where all the magic happened. This layer hosted all the business logic and aggregations, transforming raw data into actionable insights and facilitating seamless data analysis for the organization.
The comprehensive architecture we designed encompassed real-time ingestion coupled with micro-batch processing, making it a highly efficient and scalable solution for handling massive data streams. This not only improved data accessibility but also optimized data processing across the board, contributing to a more agile and data-driven decision-making process for the organization

	•	How did you handle the errors and reprocess them?
There was a try catch method in lambda. API gateway pushed the events to lambda. This lambda checked the file size and pushes it to kinesis based on file size. If there is any error while performing this task, lambda would go into exception and write this event to an s3 bucket(error files). Once the is written to s3 bucket we had an event notification enabled on the error bucket. This notification would send us an email alerting that there is a failure and we need to check the error file. Once the file/lambda issue is resolved we reprocessed the file manually by posting the event to API gateway using postman.

	•	What is the use of AWS lambda between API Gateway and Kinesis?
Data Transformation: AWS Lambda allows you to preprocess or transform data before sending it to AWS Kinesis. You can modify the data format, filter out irrelevant information, or enrich it with additional context. This can help ensure that the data ingested into Kinesis is of higher quality and relevance.
Authentication and Authorization: Lambda functions can be used to implement custom authentication and authorization logic. You can validate API requests, authenticate users, and authorize access to the Kinesis stream based on specific criteria before allowing data ingestion.
Validation and Error Handling: Lambda functions can perform input validation, ensuring that data sent to Kinesis adheres to the expected format and constraints. Additionally, they can handle errors and exceptions gracefully, providing meaningful responses to API clients and preventing malformed data from reaching Kinesis.
Data Aggregation: Lambda functions can aggregate multiple API requests into a single batch before forwarding them to Kinesis. This can help optimize resource usage and reduce the number of Kinesis records, leading to cost savings.
Enhanced Logging and Monitoring: You can implement advanced logging and monitoring within Lambda functions, allowing you to capture detailed information about incoming requests and data transformations. This visibility can aid in troubleshooting and performance monitoring.
Decoupling Components: Placing Lambda between API Gateway and Kinesis adds a layer of decoupling. This means that changes to the Kinesis stream or its processing logic won't directly impact the API Gateway or vice versa. It provides flexibility in evolving and scaling each component independently.
Custom Business Logic: You can include custom business logic within Lambda functions to perform actions beyond data transformation, such as data validation, data enrichment, and business-specific processing, before sending the data to Kinesis.

	•	While writing the data from Kinesis to S3, was there any lag? Yes or No, can you explain why?
Yes there was a lag of minimum 60 seconds. Kinesis Firehose has a minimum buffer size and buffering interval requirement, which means it waits for a minimum period (default 60 seconds) or until a certain amount of data has accumulated before it writes a file to AWS S3. This design is primarily driven by considerations related to optimizing data delivery efficiency, reducing S3 costs, and ensuring efficient S3 storage management.

Here are the key reasons for the 60-second minimum limit:
Cost Efficiency: AWS S3 charges for both storage and the number of API requests made. By aggregating data into larger files and minimizing the number of S3 API calls, Kinesis Firehose can help reduce S3 costs. Writing frequent, small files can lead to higher costs due to the overhead of API requests.
Reduced Write Overhead: Writing data to S3 involves some overhead, including authentication, network transmission, and metadata operations. By batching data and writing larger files, Firehose reduces the overhead per unit of data, making the process more efficient.
Optimized S3 Storage: Writing data in larger files can improve S3 storage efficiency. AWS S3 is designed for storing large objects, and it performs better with fewer, larger files rather than many small files. This design choice minimizes S3 fragmentation and simplifies data management.
Reduced Latency: While Kinesis Firehose aims to deliver data quickly, setting a minimum buffering interval ensures a balance between real-time data delivery and the efficiency gained by batching data. Smaller buffer sizes could lead to more frequent but less efficient writes to S3.
Network Optimization: Reducing the frequency of S3 writes can help optimize network bandwidth, especially in scenarios where data is being delivered at a high rate. Less frequent writes mean less data transfer overhead.

It's important to note that the buffering interval in Kinesis Firehose is configurable, and you can adjust it based on your specific use case and requirements. If you have a use case that demands faster delivery and can tolerate smaller file sizes, you can reduce the buffering interval to a value lower than the default 60 seconds. However, keep in mind that doing so may impact cost efficiency and S3 performance, so it should be chosen carefully based on your priorities. Conversely, if you have more flexibility in your data delivery requirements, you can increase the buffering interval for improved cost efficiency and storage optimization.



	•	How did you move the data from AWS S3 to snowflake?
We had set up a snowpipe to load data from s3 to snowflake in real time. Using Snowpipe for data ingestion offers several advantages over traditional methods. Here's why you should consider using Snowpipe when moving data from S3 to Snowflake:

Real-Time Data Ingestion: Snowpipe allows for real-time data ingestion, which means that data is automatically loaded into Snowflake as soon as new files arrive in your S3 bucket. This eliminates the need for manual intervention or scheduling data ingestion jobs.
Automation and Event-Driven Processing: Snowpipe can be configured to respond automatically to events in your S3 bucket, such as the arrival of new files. This event-driven architecture ensures that data is loaded into Snowflake as soon as it becomes available, minimizing data latency.
Scalability: Snowpipe is highly scalable and can handle large volumes of data efficiently. It automatically scales to accommodate changing data ingestion rates, ensuring that your data pipeline can handle spikes in incoming data.
Reduced Management Overhead: Traditional ETL processes often require manual scheduling and monitoring of data ingestion jobs. Snowpipe eliminates much of this management overhead by automating the data ingestion process.
Cost Efficiency: Snowpipe can help optimize costs by only processing and loading data when new files are detected. This eliminates the need for continuous resource allocation, reducing overall processing costs.
Data Freshness: Real-time data ingestion ensures that your analytics and reporting systems have access to the most up-to-date data available, enabling timely decision-making and analysis.
No Infrastructure Management: Snowpipe is a fully managed service provided by Snowflake. You don't need to provision or manage infrastructure for data ingestion, reducing operational complexity.
Data Integrity and Consistency: Snowpipe provides built-in error handling and retries, ensuring that data is ingested reliably and consistently. It also supports data validation and transformation.
Flexibility: Snowpipe supports various file formats commonly used in data storage, such as JSON, Parquet, and CSV. It can handle semi-structured and structured data, making it versatile for different data types.
